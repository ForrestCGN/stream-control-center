KURZBEWERTUNG
- Das Dashboard ist grundsätzlich modular aufgebaut: index.html lädt app.js und danach einzelne Moduldateien aus dashboard/modules/.
- Die meisten aktiven Module hängen sich an window.CGN oder reagieren auf das Event cgn:module-show.
- Es gibt aber noch keinen wirklich einheitlichen Modulstandard. Module nutzen unterschiedliche Initialisierungsmuster, unterschiedliche API-Basen, unterschiedliche Config-Strategien und unterschiedlich harte Fallbacks.
- Für neue Module ist die vorhandene Struktur nutzbar, aber vor weiterer Skalierung sollte ein verbindlicher Modulvertrag festgelegt werden.
- Auffälligster echter Fehler im Frontend: Mojibake/Encoding-Fehler in adminconfigs.js und sound.js. Texte wie fÃ¼r, GerÃ¤t, Â·, PrioritÃ¤t werden dem Benutzer falsch angezeigt.
- Ebenfalls wichtig: Backup-Dateien liegen im Dashboard-Archiv. Wenn diese in htdocs landen, sind alte JS/CSS-Stände öffentlich abrufbar und erzeugen Verwechslungsgefahr.

BOOT- UND LADEABLAUF
1. index.html lädt CSS-Dateien: app.css und je Modul eine CSS-Datei.
2. index.html definiert feste Panel-Container:
   - #sectionhomeModule => data-module-panel="sectionhome"
   - #streamdeskModule => data-module-panel="streamdesk"
   - #controlhomeModule => data-module-panel="controlhome"
   - #alertsModule => data-module-panel="alerts"
   - #obsModule => data-module-panel="obs"
   - #adminconfigsModule => data-module-panel="adminconfigs"
   - #soundModule => data-module-panel="sound_system"
   - #hugModule => data-module-panel="hug"
3. app.js erstellt window.CGN als globalen Dashboard-Kern.
4. app.js registriert harte Modul-Metadaten in window.CGN.modules und Navigationsbereiche in window.CGN.sections/moduleCatalog.
5. app.js ruft refreshAuth() auf und fragt /api/auth/session + /api/auth/status ab.
6. app.js liest localStorage für aktives Modul und aktive Section.
7. Bei Modulwechsel setzt app.js Panel sichtbar/unsichtbar und feuert cgn:module-show.
8. Die Module laden ihre Daten meist erst beim cgn:module-show oder direkt beim Mount.

GLOBALER DASHBOARD-KERN app.js
- window.CGN.activeModule: aktuell sichtbares Modul, Standard sectionhome.
- window.CGN.activeSection: aktueller Hauptbereich, Standard live.
- window.CGN.auth: Auth-Status mit authenticated, user, twitchLoginConfigured.
- window.CGN.modules: harte Liste aktiver Module mit title, panelId, group, overlayLink und reload().
- window.CGN.sections: harte Hauptnavigation Live, Control, Community, System, Admin.
- window.CGN.moduleCatalog: harte Kachel-/Modulliste inklusive vieler vorbereiteter, aber noch deaktivierter Module.
- window.CGN.favorites: aktuell alerts, hug, obs, sound_system. Wird im geprüften Frontend nicht sichtbar genutzt.
- window.CGN.api(path, options): zentrale Fetch-Hilfe. Erwartet JSON, wirft bei HTTP-Fehler oder data.ok === false einen Error.
- window.CGN.el(): kleiner DOM-Helper, im aktuellen Bestand kaum/nicht genutzt.
- window.CGN.esc(): HTML-Escape für dynamische Texte.
- setActiveSection(): setzt eine Section und öffnet sectionhome.
- setActiveModule(): schaltet Panels, Navigation, Titel, Breadcrumb, Overlay-Link und localStorage.
- reloadActiveModule(): ruft reload() des aktiven Moduls.
- refreshAuth(): liest Auth-Status und rendert Loginbox.

AUTH IM DASHBOARD

- Aktionen:

  - POST /api/auth/bootstrap-owner-local mit { displayName: "Local Owner" }

- Bewertung: Auth-Anzeige ist eingebaut, aber Frontend-seitig werden Module/Buttons noch nicht konsequent anhand von Rollen/Rechten gesperrt. Viele Texte sagen vorbereitet/gesperrt, echte Rechteprüfung muss backendseitig und in den Modulen noch sauber vereinheitlicht werden.

NAVIGATION UND MODULE
- Hauptnavigation links besteht aus festen Buttons für Sections, nicht aus Modul-Buttons.
- Die eigentlichen Modul-Kacheln werden in sectionhome.js aus window.CGN.sections und window.CGN.moduleCatalog generiert.
- app.js enthält noch Code für .nav-item[data-module]. Solche Elemente existieren in index.html nicht. Das ist vermutlich Altcode/toter Code aus der vorherigen Navigation.
- Neue Module müssen aktuell an mindestens drei Stellen eingetragen werden:
  1. index.html: CSS/JS einbinden und section-Container anlegen.
  2. app.js: window.CGN.modules ergänzen.
  3. app.js: moduleCatalog ergänzen und in sections.items einem Bereich zuordnen.
- Besser wäre später: Navigation/Modulkatalog aus config/dashboard_navigation.json oder einer Backend-Route laden. Aktuell ist diese Config im Adminbereich nur vorbereitet, app.js nutzt sie nicht.

MODULSTANDARD - IST-ZUSTAND
- Gemeinsamer Nenner: Module sind normale Browser-JS-Dateien, meist IIFE, hängen eine globale Variable an window.*Module und reagieren auf cgn:module-show.
- Es gibt keinen einheitlichen Namen für aktive Unterseiten: alerts nutzt state.page/state.modal, obs nutzt state.activeTab, sound nutzt activeSection, hug nutzt activeTab.
- Es gibt keinen gemeinsamen Modul-Base-Helper für: root finden, loading/error rendering, Tab-Handling, API-Basis, Rechteprüfung, Toast/Note, Dirty-State, Speichern/Reload.
- Das ist noch wartbar bei wenigen Modulen, wird aber bei vielen Modulen schnell unübersichtlich.

- Schreiben: nein.

2) streamdesk.js
- Zweck: Live-Bedienoberfläche für schnelle OBS-Szenen und Userinfo.
- Liest Config über GET /api/dashboard/controlcenter/streamdesk.

- Bewertung: Praktisch, aber Config-Fallback ist im Frontend hart codiert. Schreib-/Speicherfunktion für streamdesk.json gibt es im Dashboard nicht. Userinfo unterstützt mehrere Payload-Formate, was robust ist, aber die API sollte langfristig ein einheitliches Format liefern.

CONFIG LESEN UND SCHREIBEN - IST-ZUSTAND
- Es gibt keine einheitliche Config-Strategie im Dashboard-Frontend.
- app.js liest keine Dashboard-Konfiguration vom Backend. Navigation, Sections und Module sind hart codiert.
- streamdesk.js liest eine Backend-Config, fällt aber auf hart codierte Fallback-Config zurück. Kein Speichern.
- adminconfigs.js ist als zentrale Config-Ansicht vorbereitet, liest Registry und Configs über /api/dashboard/controlcenter/*, schreibt aber aktuell nur Sound-System-Spezialwerte über /api/sound/settings.
- obs.js liest/schreibt seine Config über /obs/dashboard/config. Das ist ein separater Config-Weg außerhalb /api/dashboard/controlcenter/config/:id.
- alerts.js liest/schreibt Alert-System-Config über /api/alerts/config und viele DB-nahe Alert-Ressourcen über eigene CRUD-Routen.
- sound.js liest/schreibt über /api/sound/settings und /api/sound/output. Nach Schreiben wird reload ausgelöst.
- hug.js liest nur einen zusammengefassten Dashboard-Status. Kein Config-Schreiben.

EMPFOHLENER CONFIG-STANDARD FÜR NEUE MODULE
- Jedes Modul sollte eine eigene Backend-Config-Route haben: GET /api/dashboard/<section>/<module>/config und PUT/POST /api/dashboard/<section>/<module>/config oder über den bestehenden Controlcenter-Config-Registry-Ansatz.
- Readonly-Status separat: GET /api/dashboard/<section>/<module>/status.
- Schreibende Aktionen separat: POST /api/<module>/<action>.
- Normale Configs in config/*.json oder SQLite-Settings, aber der Weg muss pro Modul dokumentiert sein.
- Secrets nie ans Frontend ausgeben. Nur masked/status/hasSecret.
- Jede Änderung später mit Auth/Rollenprüfung und Audit-Log.
- Frontend sollte keine produktiven Fallback-Configs enthalten, die zu stark vom Backend abweichen können. Fallbacks sind ok für Anzeige, aber nicht als heimlicher zweiter Source of Truth.

   - alerts.js mit über 2000 Zeilen enthält API, State, Renderlogik, Tabellen, Modals, Preview, Upload und Tests in einer Datei.

(function(){
  const root = document.getElementById('<modul>Module');
  if (!root) return;

  const state = {
    loading: false,
    error: '',
    note: '',
    data: null,
    activeTab: 'overview'
  };

  function esc(v){ return window.CGN?.esc ? window.CGN.esc(v) : String(v ?? ''); }
  async function api(path, options){ return window.CGN.api(path, options || {}); }

  function render(){
    if (!root) return;
    if (state.loading) { root.innerHTML = '<section class="card glass"><h2>Lädt...</h2></section>'; return; }
    if (state.error) { root.innerHTML = `<section class="card glass"><h2>Fehler</h2><p class="bad">${esc(state.error)}</p></section>`; return; }
    root.innerHTML = `...`;
    bindActions();
  }

  function bindActions(){
    // Eventdelegation bevorzugen, nicht nach jedem Render unkontrolliert neue Listener stapeln.
  }

  async function loadAll(force){
    if (state.loading && !force) return;
    state.loading = true;
    state.error = '';
    render();
    try {
      state.data = await api('/api/dashboard/<section>/<modul>/status');
    } catch (err) {
      state.error = err.message || String(err);
    } finally {
      state.loading = false;
      render();
    }
  }

  window.addEventListener('cgn:module-show', event => {
    if (event.detail?.module === '<modul>') loadAll(true);
  });

  window.<ModulName>Module = { loadAll };
})();
