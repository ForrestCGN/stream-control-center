BEZIEHUNGEN / WICHTIGE DATENFLÜSSE

Alert-System
- alert_types definiert, welche Alert-Typen überhaupt existieren: z.B. twitch/follow, twitch/bits, kofi/donation, tipeee/donation.
- alert_rules hängt logisch an source + type_key und optional an Schwellenwerten min_value/max_value. Regeln referenzieren Sound-Assets, Display-Profile, Textvarianten und Chat-Blöcke meist über IDs.
- alert_assets enthält hochgeladene Dateien. public_url zeigt auf /assets/..., file_path auf htdocs/assets/...; duration_ms/probe_json kommen von ffprobe.
- alert_display_profiles enthält die visuelle Darstellung. alert_events speichert zusätzlich display_settings_json als Snapshot, damit ein bereits erzeugter Alert nicht zwingend nachträglich sein Aussehen ändert.
- alert_events ist Queue + History. status zeigt den Stand. final_* Felder enthalten die fertig gerenderten Texte/Werte.
- alert_provider_events ist vorgeschaltetes Provider-Log für Ko-fi/Tipeee und enthält raw_json. Diese Tabelle ist wichtig für Deduplizierung und Debugging, kann aber sensible Rohdaten enthalten.
- alert_settings enthält Provider-Konfigurationen. In der DB liegen erkennbare Secret-/Token-/API-Key-Felder; diese sollten langfristig eher in Secrets/.env oder verschlüsselt liegen, nicht offen in SQLite.

Sound-System
- sound_settings ist eine Key-Value-Konfiguration. Aktuell liegen Queue-/Output-/Overlay-/Kategorie-/Prioritätsregeln als JSON in value_json.
- Alert-System livealert verweist in alert_settings auf /api/sound/play und koppelt Alert-Anzeige mit Sound-Start.

Dashboard/Auth/Audit
- dashboard_users ist der interne Benutzerstamm.
- dashboard_identities verknüpft externe Provider-Identitäten wie Twitch mit dashboard_users.
- dashboard_sessions speichert Sessions über token_hash, nicht Klartext-Token.
- dashboard_roles und dashboard_permissions bilden Rollen/Rechte ab. owner hat Wildcard *.
- dashboard_audit_log speichert Aktionen. Das passt zu deinem Ziel, später nachvollziehen zu können, welcher User was gemacht hat.

Hug/Rehug
- hug_users speichert pro User Gesamtzähler.
- hug_pair_stats speichert Paarstatistik Actor -> Target.
- hug_texts und hug_types bilden editierbare Text-/Typ-Basis für zufällige Hug-Ausgaben.
- hug_pending_rehugs ist die temporäre Tabelle für offene Rehug-Fenster; aktuell leer.
- hug_settings hält globale Optionen wie enabled und rehugWindowSeconds.

Tagebuch
- tagebuch_state ist der aktuelle Runtime-State: aktuelle Seite, Datum, Stream aktiv/inaktiv.
- tagebuch_runtime_events ist ein Ereignislog, keine vollständige Inhaltsspeicherung.
- tagebuch_user_stats und tagebuch_daily_user_stats speichern aggregierte Entry-Zähler. Die eigentlichen Tagebuchtexte liegen in dieser Datenbank nicht als eigene Tabelle vor.

ToDo
- todo_user_stats und todo_daily_stats speichern nur Statistik nach Autor/Ziel. Die eigentlichen ToDo-Inhalte sind in dieser DB nicht als Einzeltabelle sichtbar.

Challenge
- challenge_runtime_events enthält Laufzeitereignisse.
- challenge_user_mode_stats aggregiert User-/Mode-Statistiken.

AUFFÄLLIGKEITEN / PRÜFPUNKTE

- Keine Views vorhanden: True.
- Keine Trigger vorhanden: True.
- Leere Tabellen: hug_pending_rehugs
- alert_events.rule_id ohne passende alert_rules.id: 0
- alert_rules.sound_asset_id ohne passende alert_assets.id: 0
- alert_rules.display_profile_id ohne passende alert_display_profiles.id: 0
- alert_display_profiles mit is_default=1: 1 -> 3:testfollow
- alert_settings enthält Schlüssel/Werte mit Secret/Token/API-Key-Hinweisen: provider_kofi_webhook, provider_tipeee_socket

Konkrete Bewertung
- Die DB ist strukturell sauber lesbar; integrity_check ist ok und foreign_key_check meldet keine Fehler.
- Es gibt nur wenige echte Foreign Keys. Vieles ist logisch per IDs/JSON gekoppelt. Für spätere Module heißt das: Beim Löschen/Umbenennen im Dashboard muss der Code Referenzen sauber prüfen, weil SQLite das nicht überall erzwingt.
- Viele Einstellungen liegen als JSON in Key-Value-Tabellen. Das ist flexibel und passt zu deinem Dashboard-Ziel, ist aber weniger gut für Suche/Filter/Validierung. Für stark strukturierte Bereiche wie Alert-Regeln ist die aktuelle Mischung aus Tabellen + JSON sinnvoll.
- Provider-Rohdaten können sensible Daten enthalten. Für produktiven Betrieb sollten raw_json-Retention, Redaction und/oder Verschlüsselung fest eingeplant werden.
- Für neue Module sollte immer schema_versions genutzt werden, Migrationen idempotent per CREATE TABLE IF NOT EXISTS / ALTER TABLE defensiv, ohne bestehende Daten zu überschreiben.

====================================================================================================
EMPFOHLENER STANDARD FÜR NEUE MODULE

1. Modul-Schema
- Eigene Tabellen mit eindeutigem Prefix, z.B. hug_*, alert_*, sound_*.
- Eine Settings-Tabelle nur dann, wenn das Modul viele flexible JSON-Einstellungen braucht. Sonst lieber konkrete Spalten.
- schema_versions pro Modul setzen/prüfen.
- created_at und updated_at konsequent als ISO-Text speichern.
- enabled als INTEGER 0/1 nutzen, wenn Einträge aktiv/inaktiv schaltbar sind.
- Für User immer mindestens login + display_name speichern; wenn möglich user_id als stabilen Schlüssel nutzen.

2. Config/JSON
- JSON-Spalten nur für flexible Layouts/Provider-Payloads/verschachtelte Optionen verwenden.
- Keine Secrets im Klartext in value_json speichern. Mindestens redigieren, besser aus .env/secrets lesen und in DB nur Status/Referenz speichern.
- Bei JSON immer Default-Werte im Code mergen, damit alte Datensätze nach Updates nicht kaputt gehen.

3. Dashboard
- Änderungen an Tabellen/Settings immer in dashboard_audit_log schreiben.
- Rollen/Rechte über dashboard_permissions prüfen, nicht im Frontend vertrauen.
- Löschfunktionen bei verknüpften Daten mit Referenzprüfung bauen, weil viele Beziehungen logisch statt per FK laufen.

4. Datenbankzugriff
- Bestehende DB D:\Streaming\stramAssets\data\sqlite\app.sqlite erweitern, niemals ersetzen.
- Migrationen sanft und rückwärtskompatibel schreiben.
- Für Statistik/History separate Tabellen verwenden; keine History in Config-JSON verstecken.
- Für große Rohdaten/Events Retention einplanen, sonst wächst app.sqlite unnötig.
