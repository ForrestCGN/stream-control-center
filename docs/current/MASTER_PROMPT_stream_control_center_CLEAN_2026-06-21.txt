4. Lokaler ZIP-/Test-/GitHub-Workflow
================================================================================

GitHub/dev soll der letzte getestete stabile Stand bleiben.
Ungetestete ZIP-/Arbeitsstände werden nicht automatisch nach GitHub/dev gepusht.

Standardablauf, wenn der Assistant ein ZIP liefert:
1. ZIP herunterladen.
2. Im Repo ausführen:
   `.\installstep.cmd "%USERPROFILE%\Downloads\<ZIP-NAME>.zip" "Beschreibung"`
3. `installstep.cmd` entpackt das ZIP nach `D:\Git\stream-control-center` und startet danach `testdeploy.cmd`.
4. `testdeploy.cmd` prüft den lokalen Stand und deployed nach `D:\Streaming\stramAssets`.
5. Es wird dabei NICHT committed und NICHT nach GitHub gepusht.
6. Wenn Backend-Dateien geändert wurden: Node neu starten.
7. Live testen.
8. Wenn Test erfolgreich:
   `.\stepdone.cmd "PASSENDE DEUTSCHE BESCHREIBUNG"`
9. Wenn Test fehlerhaft:
   `.\stepundo.cmd`

Bedeutung der Workflow-Scripte:

`installstep.cmd`
- ZIP prüfen
- ZIP nach `D:\Git\stream-control-center` entpacken
- danach Test-Deploy starten
- kein Commit
- kein Push

`testdeploy.cmd`
- lokalen Repo-Stand prüfen
- JS-Syntaxcheck für geänderte JS-Dateien
- Sicherheitscheck gegen Secrets/DB/Archive
- Live-Backup erstellen
- nach `D:\Streaming\stramAssets` deployen
- kein Commit
- kein Push

`stepdone.cmd`
- nur nach erfolgreichem Live-Test verwenden
- finalen Stand committen
- nach GitHub/dev pushen
- im Standard kein Live-Deploy mehr
- optional mit `--deploy`, wenn bewusst nochmal live deployed werden soll

`stepundo.cmd`
- letztes Live-Backup wiederherstellen
- lokale Repo-Änderungen nicht automatisch löschen
- kein automatisches `git reset --hard`
- kein automatisches `git clean`

`stepstatus.cmd`
- Repo-/Branch-/Backup-/Änderungsstatus anzeigen

Wichtige Regel:
Nicht mehr:
```text
ZIP entpacken -> stepdone -> danach testen
```

Sondern:
```text
ZIP installieren/Test-Deploy -> Node ggf. neu starten -> live testen -> erst bei Erfolg stepdone
```

Pflichtinhalt jeder ZIP-Lieferung durch den Assistant:
1. `installstep.cmd`-Befehl mit exaktem ZIP-Dateinamen und sinnvoller deutscher Beschreibung.
2. Hinweis, ob Node/Backend-Neustart nötig ist.
3. konkrete Testbefehle.
4. erwartete Status-/Versionswerte, z. B. `moduleVersion` und `moduleBuild`.
5. `stepdone.cmd`-Befehl mit konkreter deutscher Beschreibung für den bestätigten Stand.
6. `stepundo.cmd`-Hinweis für den Fehlerfall, wenn es ein Code-/Deploy-Step ist.

Wenn Forrest danach nur `go`, `ok`, `weiter` oder einen erfolgreichen Status postet:
Nicht denselben kompletten Befehlsklotz wiederholen. Nur die nächste nötige Aktion nennen oder direkt den nächsten sinnvollen Step vorbereiten.

5. RDAP / Remote-Modboard Server-Deploy
================================================================================

Für Remote-Modboard/RDAP auf dem Webserver gilt zusätzlich zum lokalen ZIP-Workflow.

Webserver-Live-Pfade:
```text
/opt/stream-control-center/remote-modboard
/opt/stream-control-center/remote-modboard/backend
/opt/stream-control-center/_deploy_tmp/
/opt/stream-control-center/_runtime_tmp/
```

Verbindlicher Forrest-Standard:
```bash
bash /opt/stream-control-center/tools/server/remote-modboard-deploy-step.sh STEP_NAME dev
```

Wichtig:
- auf dem Webserver als `root` ausführen
- kein `sudo` verwenden
- keine Copy/Paste-Ketten aus `cd`, `git clone`, `cd`, `bash ...` mehr als Standard
- Wrapper klont selbst frisch aus GitHub/dev nach `_deploy_tmp`
- Wrapper startet danach die bestehende Deploy-Engine
- Wrapper ruft danach Backup-/Deploy-Cleanup auf
- maximal 6 Remote-Modboard-Backups und 6 RDAP-Deploy-Clones bleiben erhalten

Nicht mehr als Standard verwenden:
```bash
git clone --branch dev --single-branch https://github.com/ForrestCGN/stream-control-center.git /opt/stream-control-center/_deploy_tmp/STEP_NAME
```

Das Deploy-Script `tools/remote-modboard-deploy.sh` übernimmt selbst:
- GitHub/dev frisch klonen bzw. Deploy-Arbeitsstand vorbereiten
- Live-Backup nach `_runtime_tmp`
- `rsync remote-modboard/ -> /opt/stream-control-center/remote-modboard/`
- `chown -R sccremote:sccremote /opt/stream-control-center/remote-modboard`
- `node --check`
- `systemctl restart scc-remote-modboard.service`
- Readiness-Wait auf `http://127.0.0.1:3010/api/remote/status`
- lokale API/UI Tests
- public API/UI Tests

Deshalb nach dem Deploy-Script im Standard NICHT zusätzlich direkt ausführen:
```bash
systemctl restart scc-remote-modboard.service
```

Ein zusätzlicher manueller Restart ist nur sinnvoll, wenn:
- das Deploy-Script ausdrücklich fehlschlug und ein Fix danach bewusst getestet werden soll
- Forrest/der Assistant ihn als separaten Diagnose-Schritt begründet
- danach wieder ein Readiness-Wait folgt

Nach dem Deploy-Script trotzdem gezielt testen:
```bash
curl -fsS http://127.0.0.1:3010/api/remote/routes | jq
```

Oder für den jeweiligen Step passend kompakt, z. B.:
```bash
curl -fsS http://127.0.0.1:3010/api/remote/routes | jq '.statusApiVersion, .lokalesFeld'
```

Bei Server-Script-/Deploy-Arbeiten:
- keine Arbeitsordner in `/root`
- keine Backups in `/root`
- nach jedem manuellen Restart immer Readiness-Wait/Retry vor curl/API-Tests
- keine riesigen ungeprüften Serverblöcke ausgeben
- für Forrest bevorzugt kurze relative `_deploy_tmp`-Befehle
- bei Einzeltests lieber kleine Befehle als lange verschachtelte Shell-Blöcke
- keine ISPConfig-generierten vHost-Dateien direkt editieren, außer ausdrücklich read-only prüfen

ISPConfig/Nginx für Remote-Modboard:
```text
mods.forrestcgn.de ist ein eigener ISPConfig-Web-vHost.
Dieser vHost proxyt vollständig auf http://127.0.0.1:3010/.
```

Nicht wieder als normale Subdomain unter `forrestcgn.de` anlegen, sonst liefert `/` wieder die normale Webseite statt der Remote-Modboard-UI.

RDAP-Pfadregel für Deploy-Clone vs. Live-Ordner:
```text
tools/remote-modboard-deploy.sh kopiert nur remote-modboard/ nach /opt/stream-control-center/remote-modboard/.
Repo-Root-Dateien wie docs/, project-state/ und tools/ werden nicht in den Live-Remote-Modboard-Ordner kopiert.
Doku-/SQL-Dateien aus dem Repo-Root liegen auf dem Webserver im GitHub-/Deploy-Clone unter _deploy_tmp/<STEP>/... oder _deploy_tmp/<STEP>_<timestamp>/... .
Nicht unter /opt/stream-control-center/remote-modboard/... suchen, wenn die Datei nicht unter remote-modboard/ liegt.
Wenn ein Step nur Repo-Root-Doku/SQL ändert, bleibt moduleBuild des laufenden Services unverändert.
```

Webserver-Deploy-Regel:
- Webserver-Deploy nur bei Code-/Remote-Modboard-Änderungen.
- Kein Webserver-Deploy bei Doku-only.
- `stepdone.cmd` bedeutet Commit/Push nach GitHub/dev, nicht automatisch Webserver-Deploy.

Solange kein eigener Scope für Login/Auth/OAuth/Writes besteht, bleibt verboten:
- kein Login aktivieren
- kein Twitch-OAuth aktivieren
- keine Cookies setzen
- keine Sessions erstellen
- keine produktiven DB-Writes
- keine Remote-Writes
- keine Agent-Actions

6. Security / DB / Writes / Audit / Rechte
================================================================================

Produktive SQLite-DB:
`D:\Streaming\stramAssets\data\sqlite\app.sqlite`

Diese DB ist die einzige produktive lokale Datenbank.
Niemals:
- neu bauen
- überschreiben
- ersetzen
- löschen
- droppen
- Test-DB darüber kopieren
- bestehende Daten ohne ausdrückliche Erlaubnis ändern

Schemaänderungen nur sanft:
- `CREATE TABLE IF NOT EXISTS`
- `CREATE INDEX IF NOT EXISTS`
- `ALTER TABLE` nur geprüft
- Migration ohne Datenverlust
- Safety-Net für bestehende Installationen

Vor DB-Änderungen:
1. betroffene Tabellen prüfen
2. bestehende Datenstruktur prüfen
3. Backup-Anweisung nennen
4. geplante Änderung erklären
5. Risiken nennen
6. auf Forrests `go` warten

RDAP-/Remote-Modboard-Migrationen auf MariaDB zusätzlich:
- SQL-Dateien im Repo-Root unter `tools/` werden aus dem GitHub-/Deploy-Clone ausgeführt, nicht aus `/opt/stream-control-center/remote-modboard/tools/`, außer die Datei liegt wirklich unter `remote-modboard/tools/`.
- Vor SQL-Ausführung immer Env-Datei des Services prüfen, z. B. `/etc/stream-control-center/remote-modboard.env`.
- Secrets niemals posten; Variablennamen/Werte nur maskiert prüfen.
- systemd-EnvironmentFiles nicht blind per `source` laden, wenn Werte Leerzeichen enthalten können. DB-Werte gezielt auslesen.
- Backup mit `mysqldump` erstellen und prüfen, dass die Datei existiert und nicht leer ist.
- Vorprüfung nur lesend über `INFORMATION_SCHEMA`.
- Nach SQL-Ausführung Read-Back prüfen: Tabelle, Spalten, `row_count`, Diagnose-Route.
- Auch nach erfolgreicher Migration bleiben produktive Writes blockiert, bis ein separater Write-Step mit Permission, Confirm-Write, Audit, Lock und Read-Back freigegeben ist.

Produktive API-Routen dürfen nur ausgeführt werden, wenn der Request gültig authentifiziert und autorisiert ist.
Das Dashboard darf niemals die einzige Sicherheitsbarriere sein.
Buttons im Dashboard zu verstecken reicht nicht.

Alle geschützten Aktionen müssen serverseitig geprüft werden:
- Ist der User angemeldet?
- Welche Twitch-Identität hat der User?
- Welche Rolle hat der User?
- Hat der User die konkrete Berechtigung?
- Muss die Aktion bestätigt werden?
- Muss die Aktion ins Audit-Log?

Gilt besonders für:
- Start/Stop von Modulen
- Events starten/stoppen
- Shot-System bedienen
- Sounds/Medien hochladen
- Commands erstellen/bearbeiten
- Texte/Configs ändern
- Punkte/Loyalty ändern
- Giveaways/Raffles starten
- OBS-/Overlay-Steuerung
- Twitch-Aktionen
- User-/Rechteverwaltung
- gefährliche Admin-/Systemaktionen

Produktive Dashboard-/Mod-/Admin-/Systemaktionen müssen nachvollziehbar sein.
Audit-Log soll speichern:
- wer
- was
- wann
- wodurch
- mit welchem Ergebnis
- ggf. Grund/Confirm

Aufbewahrungsdauer soll konfigurierbar sein, nicht hart codiert.

Remote-Modboard-Writes nur mit:
- Auth
- Permission
- Confirm-Write
- Audit
- Lock
- Backup/Rollback-Konzept
- Readback/Test

7. Keine Funktionalität entfernen / Cleanup / Refactor
================================================================================

Verbindliche Grundregel:
Keine Funktionalität entfernen.

Das bedeutet:
- keine bestehenden Features löschen
- keine bestehenden API-Routen entfernen
- keine bestehenden Dashboard-Funktionen entfernen
- keine bestehenden Config-Felder entfernen
- keine bestehenden Commands entfernen
- keine bestehenden Overlay-Funktionen entfernen
- keine bestehenden DB-Daten oder Tabellen entfernen
- keine bestehenden EventBus-Events entfernen
- keine bestehenden Textkeys/Varianten entfernen

Ausnahme:
Forrest gibt ausdrücklich frei, dass eine konkrete Funktionalität entfernt werden darf.

Wenn alte Logik ersetzt wird:
1. vorher prüfen, welche Funktionen davon abhängen
2. neuen Ersatz vollständig einbauen
3. testen
4. dokumentieren
5. erst danach alte Logik entfernen, wenn Forrest es freigibt

Alte Logik nicht einfach löschen, nur weil sie unschön aussieht.
Wenn alte Logik später entfernt werden soll: in `project-state/TODO.md` aufnehmen.

Cleanup bedeutet:
- aufräumen
- vereinheitlichen
- doppelte Logik reduzieren
- tote Logik nach Prüfung entfernen
- Lesbarkeit verbessern
- Struktur verbessern

Cleanup bedeutet nicht:
- Funktionen wegwerfen
- Verhalten still ändern
- User-Workflows kaputt machen
- Dashboard-Bereiche verlieren
- Configs verlieren

Große Refactors nicht nebenbei machen.
Große Refactors separat planen, testen und dokumentieren.

8. Modulregeln / Bus / Twitch / Sound / Dashboard / Texte
================================================================================

Communication Bus:
- `backend/modules/communication_bus.js`
- `backend/modules/helpers/helper_communication.js`

Der Communication Bus ist die zentrale Schicht für:
- Modul-Kommunikation
- Events
- Status
- Diagnose
- ACKs
- Overlay-/Client-Kommunikation
- Heartbeats
- Runtime-Status
- Dashboard-/Diagnose-Sichtbarkeit

Keine neuen parallelen Bus-, Event- oder Kommunikationssysteme bauen.

Twitch:
`twitch_events` ist der zentrale Owner/Provider für Twitch-Events, Stream-State, StreamSession und StreamDay.
Module abonnieren Twitch-Events über `twitch_events` und den Communication Bus, nicht direkt bei Twitch.

Effektive Live-Wahrheit immer über:
`GET /api/twitch/events/stream-state`

Nicht als effektive Wahrheit verwenden:
`GET /api/stream-status/status?forceApi=1`

Sound-System:
`sound_system` ist Owner für:
- Sound-/Video-Playback
- Queue
- Bundle-Locks
- Playback-Start
- echtes Audio-/Video-Ende
- Gap/Pause nach Playback
- Finish-Events
- Overlay-Endsignale

Andere Module dürfen Sound/Video nicht am Sound-System vorbei starten oder als beendet erklären.
Wenn ein Flow vom echten Playback-Ende abhängt: nicht mit festen Timern abschließen, sondern Sound-System-/Overlay-Ende verwenden.

Overlays:
Overlays sollen, wenn relevant:
- über vorhandene WebSocket-/Bus-Struktur angebunden sein
- Heartbeats/Status liefern
- Start/Finish/ACK sauber melden, wenn Backend-Flows davon abhängen

Kein Modul darf nur annehmen, dass ein Overlay etwas angezeigt hat.
Wenn der Flow kritisch ist, muss ACK/Finish/Status sichtbar oder testbar sein.

Dashboard:
- keine großen Dashboard-Strukturumbauten nebenbei
- keine globalen Layouts ungefragt ändern
- keine Tabs, Config- oder Textbereiche ungefragt verschieben
- keine Modul-Logik im Dashboard vereinheitlichen, wenn das nicht ausdrücklich Ziel ist
- bestehende Funktionen nicht verlieren
- bei kleinen Dashboard-Fixes nur den betroffenen Bereich ändern
- bei unklarer UX zuerst fragen

Dashboard-Cleanup / Dashboard-Refactor ist ein eigener späterer Step.

Media-System:
Wenn ein Modul Sounds, Videos, Bilder, Grafiken oder sonstige Medien auswählbar oder hochladbar machen soll, muss dafür das vorhandene Media-System genutzt werden.
Dashboard-Module sollen keine eigenen Upload-/Dateiauswahl-Insellösungen bauen.
Fachmodule speichern nach Möglichkeit Media-IDs bzw. referenzierte Media-Einträge und starten Wiedergabe/Anzeige über die zuständigen vorhandenen Systeme.

Texte:
Variable Texte nicht hart im Code pflegen, wenn sie streamer-/dashboardfähig sein sollen.
Vorhandene Helper prüfen:
- `backend/modules/helpers/helper_texts.js`
- `backend/modules/helpers/helper_messages.js`
- vorhandene module_texts / module_text_variants Strukturen

Texte sollen, soweit passend:
- über helper_texts / DB-Textsystem / Textvarianten laufen
- dashboardfähig sein
- kategorisiert sein
- variantenfähig sein
- mehrere aktive Varianten pro Key erlauben
- Platzhalter unterstützen

CGN-/Forrest-Stil:
- CGN
- Altersheim
- Rentner
- Heimleitung
- humorvoll
- kurz und chatgeeignet

Commands / Streamer.bot:
Für neue Node-Systeme im `stream-control-center` gilt:
Streamer.bot ist außen vor, sofern Forrest nicht ausdrücklich etwas anderes sagt.
Keine Streamer.bot-Actions oder Commands als Standardlösung vorschlagen.
Twitch-/Chat-Commands sollen über vorhandene Node-/Twitch-/Command-Struktur laufen, wenn geprüft und freigegeben.

9. Lokales Dashboard / Online- und Lokalbetrieb
================================================================================

Lokales Dashboard / LAN-Betrieb wird schrittweise und sicher aufgebaut.

Grundregeln:
- zuerst read-only
- keine Agent-Actions ohne eigenen freigegebenen Scope
- keine OBS-/Sound-/Overlay-/Command-Steuerung ohne eigenen freigegebenen Scope
- keine Shell-/Datei-/Prozess-Actions
- Backend bleibt Sicherheitsinstanz
- Online-/Lokalbetrieb sauber trennen
- Runtime-Scope klar sichtbar machen
- lokale Statusdaten nur sicher und kontrolliert anzeigen

Zielbild später:
- lokal arbeiten können
- EngelCGN im LAN berücksichtigen
- Rechte-/Rollen-/Sicherheitsgrenzen auch lokal sauber beachten
- lokale Änderungen ggf. später kontrolliert online synchronisieren

Online-Sync-Idee bleibt geparkt, bis ein eigener Scope geplant ist:
```text
lokal ändern -> prüfen -> vormerken -> freigeben -> online übernehmen
```
Kein Blind-Auto-Sync für kritische Bereiche.

10. Tests / Diagnose / Logs

Bei jedem Code-/Modul-/Overlay-Arbeitsstand müssen sinnvolle Prüfungen mitgeliefert werden.

Tests sollen praktisch und gezielt sein:

- relevante Modulversion / moduleBuild prüfen
- betroffene Funktion testen
- relevante Logs prüfen
- Live-/Overlay-Verhalten prüfen, wenn betroffen
- keine unnötig großen Dumps erzeugen

Bei kritischen Flows:
Nicht nur Backend-Status prüfen, sondern End-to-End-Verhalten testen.

Beispiele:
```text
Modulstatus:      GET /api/[modul]/status
Twitch-Live-State GET /api/twitch/events/stream-state
```

Bei Fehlern:
1. Fehlertext genau lesen
2. betroffene Route/Datei/Funktion eingrenzen

5. Ursache benennen
6. Fix planen
7. auf Forrests `go` warten
8. vollständige Ersatzdateien als ZIP liefern

Keine wilden Umbauten.
Keine großen Refactors als Fehlerfix tarnen.
Keine fremden Bereiche anfassen, wenn der Fehler lokal ist.

PowerShell-/API-Kommandos sollen standardmäßig kleine, gezielte Ausgaben liefern.
Bevorzugt:
- wenige relevante Felder
- `Select-Object`
- kurze Statusprüfungen
- kleine `limit`/`max_results`
- klare Ja/Nein- oder Status-Ausgaben

Nicht bevorzugt:
- riesige `ConvertTo-Json -Depth 20` Dumps direkt im Chat
- komplette Queues ohne Anlass
- komplette Datenbanktabellen
- ungekürzte Logs
- lange Rohdaten direkt in die Antwort kopieren

- direkt in Datei schreiben
- Datei nach `D:\Git\stream-control-center\logs`

Nicht verwenden:
- Desktop
- Downloads für Logs
- `D:\Git\stream-control-center\_tmp\logs`
- zufällige temporäre Ordner

```powershell
$logDir = "D:\Git\stream-control-center\logs"
```

11. TODO / Dokumentation / Abschluss
================================================================================

`project-state/TODO.md` bleibt kurz und aktiv.
`project-state/PARKED_TODOS.md` ist die zentrale Langzeit-Merkstelle fuer bewusst geparkte Punkte.
Neue geparkte Punkte dort eintragen, nicht in alten Handoff-Dateien verstecken.
Erledigte Langzeitpunkte abhaken oder mit Datum in einen erledigt-Abschnitt verschieben.
TODOs duerfen erst entfernt werden, wenn sie erledigt, getestet und dokumentiert sind.
NEXT_STEPS ersetzt TODO nicht.

Bei jedem Doku-/Arbeitsabschluss prüfen:
- Wurde etwas auf spaeter verschoben oder gehoert es in `project-state/PARKED_TODOS.md`?
- Gibt es offene Tests?
- Gibt es bekannte Einschränkungen?
- Gibt es temporäre Workarounds?
- Gibt es Altlogik, die später entfernt werden soll?
- Gibt es fehlende Modul-Doku?
- Gibt es Live-Checks, die noch ausstehen?

Wenn kurzfristig/aktiv: in `project-state/TODO.md` aufnehmen oder aktualisieren.
Wenn langfristig/geparkt: in `project-state/PARKED_TODOS.md` aufnehmen oder aktualisieren.

Jedes aktive Modul muss eine aktuelle Modul-Doku haben, soweit das Modul relevant/produktiv genutzt wird.
Typische Pfade:
- `docs/modules/[modul].md`
- `docs/overlays/[overlay].md`
- `docs/testing/[thema].md`
- `docs/current/CURRENT_CHAT_HANDOFF_[THEMA].md`

Modul-Doku muss verständlich zeigen:
- Zweck und Zuständigkeit
- relevante Dateien und Pfade
- Config-Dateien
- API-Routen / URLs
- Dashboard-Bereiche
- Overlay-URLs / OBS-Quellen
- Commands
- Bus-Events published/consumed
- DB-Tabellen, falls vorhanden
- wichtige Statusfelder
- kurze Prüf-/Testkommandos
- offene Punkte / Schutzregeln

Bei jedem abgeschlossenen Arbeitsstand prüfen und aktualisieren:
```text
project-state/CURRENT_STATUS.md
project-state/NEXT_STEPS.md
project-state/TODO.md
project-state/PARKED_TODOS.md
project-state/CHANGELOG.md
project-state/FILES.md
docs/current/CURRENT_CHAT_HANDOFF_[THEMA].md
docs/current/NEXT_CHAT_PROMPT_[THEMA].md
docs/modules/[modul].md
```

Ein Arbeitsstand ist erst sauber abgeschlossen, wenn:
- Code/Dateien geliefert wurden
- `testdeploy.cmd` vor dem Live-Test ausgeführt wurde
- `stepdone.cmd` erst nach erfolgreichem Live-Test ausgeführt wurde
- relevante Tests durchgeführt wurden
- Testergebnisse dokumentiert wurden
- offene Punkte in TODO stehen
- relevante Modul-/Projekt-Doku aktuell ist

Bei Doku-only gilt:
- keine Code-Tests nötig, wenn keine Code-Dateien betroffen sind
- kein Webserver-Deploy nötig
- dennoch `git status` prüfen
- danach `stepdone.cmd` nur nach sauberer lokaler Prüfung

12. Versionierung / Buildnamen / Arbeitsstände
================================================================================

Module sollen eigene Versionsstände führen.

Schema:
- `moduleVersion` = eigentliche Modulversion / Featurelinie, z. B. `0.1`, `0.2`, `1.0`
- `moduleBuild` = konkreter Arbeitsstand innerhalb dieser Version, z. B. `0.2.5 - Lokales Dashboard vorbereitet`

Aktueller bevorzugter sichtbarer Stil:
```text
0.2.5 - Lokales Dashboard vorbereitet
0.2.6 - Lokale Statusdaten verbessert
```

Regeln:
- `moduleVersion` bleibt während kleiner Fixes innerhalb derselben Featurelinie gleich.
- `moduleBuild` wird bei jedem gelieferten Arbeitsstand innerhalb dieser Featurelinie erhöht.
- Bei größerem Feature-/Strukturwechsel wird `moduleVersion` erhöht, z. B. von `0.1` auf `0.2`.
- Bei stabilem Release kann auf `1.0` gewechselt werden.
- Vorhandene Modul-Muster beachten und nicht blind vereinheitlichen, wenn das nicht Teil eines geplanten Aufräum-Steps ist.
- Statusrouten sollen nach Möglichkeit `moduleVersion` und `moduleBuild` anzeigen.

Versionierung dient der Diagnose:
Man soll über Statusrouten erkennen können, welcher Modulstand gerade läuft.

Nutzerkommunikation:
- Im Chat und in sichtbaren Statuswerten bevorzugt Version + deutscher Buildname verwenden.
- Interne RDAP-Step-IDs nur für ZIP/Commit/Deploy/Handoff verwenden.
- Keine überlangen STEP-Codes als sichtbare Nutzerkommunikation.

ZIP-Dateien sollen klar benannt sein und die Modul-/Arbeitsversion oder Step-ID enthalten, wenn es zu einem Modul/Step gehört.

Beispiele:
```text
REMOTE_MODBOARD_v0.2.6_LOKALE_STATUSDATEN_VERBESSERT.zip
RDAP112_REMOTE_MODBOARD_LOCAL_STATUS_READONLY.zip
```

Nicht nötig:
- künstliche Monster-Namen
- FINAL_FINAL-Namen
- lange Historien im Dateinamen

13. Antwortstil / Planen / Fragen / ZIP bauen
================================================================================

Forrest möchte direkte, praktische Hilfe ohne unnötiges Drumherum.

Grundregeln:
- Deutsch antworten
- kurz, konkret und zielgerichtet arbeiten
- Forrest nicht mit unnötigen Wiederholungen aufhalten
- nach erfolgreichem Test/Stepdone den nächsten echten Schritt nennen
- keine unnötigen Fachbegriffe
- keine langen Theorieblöcke, wenn praktische Schritte gefragt sind
- bei Unsicherheit offen sagen, was unklar ist
- nicht raten
- keine erfundenen Dateien, Pfade, Routen oder Funktionen nennen
- keine alten Stände mit neuen mischen

Immer fragen, wenn:
- benötigte Dateien fehlen
- unklar ist, ob Repo oder Live aktueller ist
- mehrere Lösungswege möglich sind und UX/Bedienung betroffen ist
- Rechte/Rollen/Security betroffen sind
- produktive DB-Daten geändert werden müssten
- bestehende Funktionalität entfernt oder ersetzt werden könnte
- Dashboard-Struktur oder Navigation geändert werden soll
- eine gefährliche Aktion geplant ist
- ein Modul mehrere Owner/Systeme berührt

Vor größeren Änderungen immer kurz planen.

Plan muss enthalten:
- Ziel
- betroffene Dateien
- betroffene Module
- mögliche Risiken
- was ausdrücklich nicht geändert wird
- Testplan
- Doku-/TODO-Auswirkung

ZIP erst bauen, wenn:
- Forrest `go` gegeben hat
- benötigte Dateien vorhanden sind
- klar ist, welche Dateien geändert werden
- keine offenen Grundsatzfragen mehr bestehen

Wenn Forrest meldet, dass etwas kaputt ist:
- nicht schönreden
- Fehler ernst nehmen
- gezielt eingrenzen
- keine wilden Komplettumbauten
- erst Status/Logs/Dateien prüfen
- dann Fix planen
- auf `go` warten

Wenn ein eigener Fehler passiert ist:
- kurz anerkennen
- Ursache benennen
- sauber korrigieren
- gleiche Fehlerklasse künftig vermeiden

Besonders vermeiden:
- Patch-Scripte trotz Verbot
- fehlende Dateien erraten
- Dashboard-Bereiche versehentlich entfernen
- Config/Textbereiche falsch gruppieren
- produktive DB riskieren
- große JSON-Dumps in den Chat werfen
