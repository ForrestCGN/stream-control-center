7. EventBus / Communication Bus – SEHR WICHTIG
================================================================================

Der EventBus / Communication Bus ist ein zentraler Projektbestandteil.

Wichtige Datei:

backend/modules/communication_bus.js

Wichtige Doku:

docs/system-inspection/COMMUNICATION_BUS_CONTRACT_CONSOLIDATION.md

Der EventBus ist die zentrale Schicht für:

Events
Status
Diagnose
ACKs
Overlay-/Modul-Kommunikation
Recovery-Diagnose

Neue EventBus-Anbindungen müssen:

klein und nachvollziehbar sein
rückwärtskompatibel sein
bestehende produktive Flows zunächst nur ergänzen
sauber testbar sein
keine bestehenden Systeme ungeprüft ersetzen

Der EventBus darf produktive Abläufe nicht ersetzen, solange das nicht ausdrücklich geplant, umgesetzt, getestet und freigegeben ist.

Gute Reihenfolge bei EventBus-Arbeiten:

1. Read-only Status / Diagnose ergänzen
2. Smoke-Test-Route ergänzen
3. Status-/Reset-Routen prüfen
4. Live-Test mit echtem Flow
5. Erst danach produktive Bus-Ausgabe planen

EventBus-Events sollen vorhandene Muster aus dem Repo verwenden. Keine Eventnamen frei erfinden.

Typische Actions / Zustände:

accepted
queued
started
running
finished
failed
skipped
blocked
module.started
module.finished
module.failed
module.queue.updated
module.status
module.event

15a. Neue Module / Diagnose-Registry – VERBINDLICH AB CAN-42.32
================================================================================

Bei jedem neuen Backend-Modul oder größerer Modul-Erweiterung muss direkt geprüft werden:

1. Gibt es bereits ein Modul, Helper, Bridge oder eine vorhandene Datei dafür?
2. Ist das neue Modul fachlich diagnosefähig?
3. Braucht es eine Statusroute, bevorzugt `GET /api/<modul>/status`?
4. Liefert es Standardfelder wie `module`, `moduleVersion`, `moduleBuild`, `version`, `enabled`, `routeCount`?
5. Liefert es einen standardisierten `diagnostics`-Block?
6. Muss es in die zentrale Registry `GET /api/diagnostics/registry`?
7. Bleibt die Registry-Coverage danach sauber?
8. Welche Doku-Dateien müssen gleichzeitig aktualisiert werden?

Standard-`diagnostics`-Block, soweit passend:

ok
health
module
version
build
schemaVersion
schemaReady
lastError
counts
database
state
queue
runtime
warnings
errors

Wichtige Regel:

Keine neuen Dashboard-Diagnose-Extra-Dateien pro Modul anlegen.
Keine neuen Dateien wie `*_readonly_diagnostics.*`, `*_diagnostics_ext.*` oder Spezial-Patcher, außer das wurde vorher ausdrücklich geplant und freigegeben.

Zentrale Diagnose-Struktur:

backend/modules/diagnostics.js
  -> `/api/diagnostics/registry` und Diagnose-Registry

htdocs/dashboard/modules/diagnostics.js
  -> zentrale Dashboard-Diagnose, Auswahl, Registry-Anzeige, Standard-Diagnose

htdocs/dashboard/modules/diagnostics.css
  -> Styling

Pflichttest nach neuen Diagnose-/Registry-Einträgen:

$r = Invoke-RestMethod "http://127.0.0.1:8080/api/diagnostics/registry"
$r.coverage | Select-Object ok,registryEntries,loadedModules,coveredLoadedModules,missingLoadedModules,registryOnlyEntries
