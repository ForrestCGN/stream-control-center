==========================================================================================
1. KURZFAZIT
==========================================================================================
Dieses Archiv enthält den Overlay-Bestand, aber keine vollständige Dashboard-Codebasis.
Nicht enthalten sind insbesondere typische Dashboard-Dateien wie htdocs/dashboard/index.html, dashboard-modul-js/css, Backend-Module unter backend/modules oder Config-Dateien unter config/.
Damit kann der Overlay-/Client-Aufbau vollständig aus diesem ZIP dokumentiert werden; das eigentliche Schreiben/Lesen von Configs im Backend kann hier nur über die verwendeten Client-Endpunkte abgeleitet werden.

Wichtigster Befund: Die Overlays sind historisch gewachsen und nicht einheitlich aufgebaut. Es gibt mehrere Generationen gleicher Overlays, direkte localhost-URLs, relative /api-Endpunkte, mehrere WebSocket-Implementierungen, alte Backup-/Alt-Dateien und sehr große monolithische HTML-Dateien. Für neue Module sollte ein einheitliches Pattern festgelegt werden.

Konkrete Auffälligkeiten:
- Keine zentrale Overlay-Client-Library, obwohl sich WebSocket, Fetch, Reconnect, Escape, Statusanzeige und Cache-Busting mehrfach wiederholen.
- Gemischte API-Basis: teils relative /api/..., teils http://localhost:8080/..., teils http://127.0.0.1:8080/....
- WebSocket teils dynamisch über location.host, teils hart auf localhost/127.0.0.1 gesetzt.
- Mehrere Versions-/Backup-Dateien liegen direkt produktiv neben den aktiven Overlays: .bak, .alt, .new.
- Start/Pause/Alert-Overlays sind sehr groß und enthalten CSS, HTML und JS komplett in einer Datei. Wartung ist dadurch fehleranfällig.
- Einige Overlays laden externe Fonts/Skripte/CDNs oder Twitch-GQL direkt aus dem Browser. Für OBS-Browserquellen ist das unnötig fragil.
- Fireworks ist ein Fremd-/Legacy-System mit eigener localStorage-Konfiguration und eigenen alten WebSocket-Zielen.

==========================================================================================
7. EINHEITLICHKEIT / UNGEREIMTHEITEN
==========================================================================================
- Fehlende Dashboard-Dateien: Das Archiv enthält keine echte Dashboard-Modulstruktur. Damit fehlen UI-Seiten, Formularlogik, Config-Save-Routen und Backend-Middleware.
- Backup-Dateien im Produktivordner: Mehrere .bak/.alt/.new Dateien liegen neben aktiven Dateien. Das erhöht die Gefahr, in OBS oder beim Editieren die falsche Datei zu nutzen.
- Hartcodierte localhost-URLs: Einige Dateien nutzen http://localhost:8080 oder ws://localhost:8080 statt relativer URLs/location.host. Das bricht bei Cloudflare, anderem Hostnamen oder anderer OBS-URL.
- Gemischte WebSocket-Strategien: Manche Overlays nutzen location.host, andere 127.0.0.1, andere localhost, andere normalizeWsUrl. Das sollte vereinheitlicht werden.
- Monolithische HTML-Dateien: Start, Pause und Alerts enthalten sehr viel CSS/JS direkt im HTML. Fehlerkorrekturen und Wiederverwendung sind unnötig schwer.
- Doppelte Chat-Logik: Start und Pause haben sehr ähnliche Chat-WebSocket-/Snapshot-/Render-Logik. Diese sollte als shared overlay client ausgelagert werden.
- Doppelte Escape-/Fetch-/CacheBust-Helper: escapeHtml, safeJson, cacheBust, wsUrl, dlog usw. sind mehrfach vorhanden.
- Direkte externe Abhängigkeiten: Fonts/CDNs/Twitch-GQL werden teilweise direkt aus dem Overlay geladen. Für OBS ist lokale Auslieferung stabiler.
- Legacy Fireworks: Fireworks bringt eigene Config, localStorage, Audio-Blobs und alte WS-Ziele mit. Sollte klar als Legacy/externe Komponente abgegrenzt werden.
- Clip-Overlay nutzt Twitch GraphQL direkt: Das sollte besser über das Backend laufen, damit Tokens/Client-ID/Fehlerhandling zentral bleiben.
- Uneinheitliche Benennung: Dateien nutzen gemischte Namen mit Unterstrich, v2, alt, new, Datum, Leerzeichen und Umlauten. Für produktive Quellen sollte es klare Namen geben.
- Keine klare aktive Datei-Markierung: Aus dem Ordner allein ist nicht eindeutig erkennbar, welche Version in OBS aktiv ist. Aktive Quellen sollten in einer Strukturdatei dokumentiert werden.

==========================================================================================
8. EMPFOHLENER STANDARD FÜR NEUE OVERLAY-/DASHBOARD-MODULE
==========================================================================================
Neue Module sollten sich an folgendem Zielaufbau orientieren:

Backend:
- backend/modules/<module>.js
- registriert Routen unter /api/<module>/...
- nutzt vorhandene Helper für Config, Response, Security, State, Logging und ggf. SQLite
- Initialstatus: GET /api/<module>/status
- Overlay-Zustand: GET /api/<module>/state
- Konfiguration: GET /api/<module>/config und POST /api/<module>/config
- Test/Trigger: POST /api/<module>/test oder /api/<module>/enqueue
- WebSocket-Broadcast bei Änderungen: { op: "<module>", type/reason: "state_changed", data: ... }

Dashboard:
- htdocs/dashboard/modules/<module>.js
- htdocs/dashboard/modules/<module>.css
- liest Config per GET /api/<module>/config
- speichert Config per POST /api/<module>/config
- zeigt Status per GET /api/<module>/status
- keine hartcodierten Pfade; API-Basis relativ oder über zentralen Dashboard-Helper
- Audit-/Logging-Hook für Änderungen vorbereiten

Overlay:
- htdocs/overlays/_overlay-<module>.html
- Initialzustand per GET /api/<module>/state
- Live-Updates per WebSocket
- keine Secrets, Tokens oder direkten Fremd-APIs im Browser
- relative URLs verwenden
- gemeinsame Helper für wsUrl, safeJson, escapeHtml, cacheBust und reconnect nutzen

Config/DB:
- Kleine, selten geänderte Moduleinstellungen können in config/<module>.json liegen.
- Laufzeitdaten, Queues, Historien, Statistiken und userbezogene Daten gehören in data/sqlite/app.sqlite.
- Später MariaDB mitdenken: SQL/Migrationen nicht unnötig SQLite-spezifisch bauen.
- Keine zweite Datenbank anlegen. Die aktive app.sqlite unter data/sqlite/app.sqlite bleibt Single Source of Truth.
